## Design Doc: Система авторизации (Twitch/VK/YT) и сессий

## Стек: Rust (Axum), SQLite (SQLx), SvelteKit.

## 1. Архитектура базы данных (SQLite)

Для поддержки мультипровайдерности и стабильных сессий используем три таблицы.

## Таблица users

Хранит глобальный профиль пользователя на вашем сайте.

- id (TEXT, PK): Внутренний UUID.
- display_name (TEXT): Имя для отображения в чате.
- avatar_url (TEXT): Ссылка на текущий аватар.
- created_at (DATETIME): Дата регистрации.

## Таблица user_identities

Связывает внутренний аккаунт с внешними сервисами.

- provider (TEXT): Название (twitch, vk, youtube).
- provider_user_id (TEXT): ID пользователя в этой соцсети.
- user_id (TEXT, FK): Ссылка на users.id.
- Уникальный индекс: (provider, provider_user_id).

## Таблица sessions

Хранит данные для долгого входа.

- token (TEXT, PK): Случайный длинный хеш (64+ символа).
- user_id (TEXT, FK): Чья сессия.
- expires_at (DATETIME): Когда сессия станет невалидной.

---

## 2. Логика авторизации (OAuth Flow)

1.  Redirect: Пользователь уходит на Twitch/VK.
2.  Callback: Сервис возвращает временный code на ваш бэкенд (Axum).
3.  Exchange: Axum меняет code на access_token (через reqwest).
4.  Fetch Profile: Axum запрашивает данные профиля (id, name, avatar) у провайдера.
5.  Upsert:

- Ищем в user_identities по паре provider + provider_id.
  - Если найден: обновляем display_name и avatar_url в таблице users.
  - Если не найден: создаем новую запись в users, затем привязываем её в user_identities.

6.  Session Start: Генерируем новый случайный токен, пишем в sessions на 30 дней.
7.  Set-Cookie: Отправляем httpOnly, Secure, SameSite=Lax куку в браузер.

---

## 3. Управление сессиями (Sliding Expiration)

Чтобы пользователь не "вылетал" каждые 30 дней, реализуем продление:

- Порог обновления: 15 дней (половина срока).
- Логика: При каждом запросе к API:

1. Если до expires_at осталось < 15 дней. 2. UPDATE sessions SET expires_at = now + 30 days. 3. Отправить новый Set-Cookie с тем же токеном, но обновленным Max-Age.

---

## 4. Безопасность

- HttpOnly куки: Обязательно, чтобы JavaScript (SvelteKit на клиенте) не имел доступа к токену.
- CSRF Protection: При OAuth используйте параметр state (случайная строка, проверяемая при возврате).
- Cleanup Task: Раз в сутки удалять просроченные сессии:
  DELETE FROM sessions WHERE expires_at < CURRENT_TIMESTAMP;

## Процесс: Слияние временного профиля с основным

Используется при переходе пользователя из состояния «Аноним с Push-уведомлениями» в состояние «Авторизованный пользователь».

1.  Триггер: Успешный ответ от OAuth-провайдера (Twitch/VK) при наличии у пользователя активной «анонимной» сессии.
2.  Проверка конфликта:

- Если provider_user_id не найден в БД: Текущий анонимный профиль обновляется (is_anonymous = false, заполняется display_name, создается запись в user_identities).
  - Если provider_user_id уже существует (у юзера есть старый аккаунт): Запускается процедура слияния (Merge).

3.  Логика Merge (Транзакция):

- Перенос данных: Все связанные записи из push_subscriptions перепривязываются к ID старого аккаунта (UPDATE ... SET user_id = permanent_id).
  - Удаление дублей: Если один и тот же девайс (endpoint) есть в обоих аккаунтах, приоритет отдается актуальной подписке, дубль удаляется.
  - Утилизация: Временный анонимный профиль и его сессия удаляются из таблиц users и sessions.

4.  Результат: Пользователь получает обновленную куку сессии, привязанную к его постоянному ID, при этом Push-уведомления, настроенные ранее, продолжают работать.

---

⚓ Ключевое правило: Слияние должно происходить атомарно (внутри одной SQL-транзакции), чтобы исключить потерю данных Push-подписки при сбое.
